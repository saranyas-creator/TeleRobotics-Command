# Command Worker

## Overview

The **Command Worker** is responsible for exchanging command messages between the Qt application and the Robot Software.

The worker inherits from **ZmqWorkerBase** and runs in its own worker thread managed by the **ZmqManager**. It communicates with the Router Service using a **ZeroMQ DEALER** socket. After registering as **UI_CMD**, the worker sends user commands to the Robot Software and continuously receives acknowledgements, responses, and status messages.

Unlike the Camera Worker and Watchdog Worker, the Command Worker supports **bidirectional communication**. It both transmits commands generated by the Qt application and receives responses from the Robot Software.

### Responsibilities

- Register with the Router Service as **UI_CMD**
- Send command messages to the Robot Software
- Receive acknowledgements and command responses
- Parse incoming JSON messages
- Forward processed messages to the Qt application

---

# Architecture

The Command Worker acts as the communication interface between the Qt application and the Robot Software.

```text
                 Qt Application
                        │
        ┌───────────────┴───────────────┐
        │                               │
        ▼                               ▼
   ZmqManager                  Command Widgets
        │                               ▲
        ▼                               │
   Command Worker ──────────────────────┘
```

### Component Responsibilities

- **ZmqManager**
  - Creates and manages the Command Worker.
  - Starts and stops the worker thread during the application lifecycle.

- **Command Worker**
  - Sends command messages to the Robot Software.
  - Receives acknowledgements and responses.
  - Parses incoming JSON messages.
  - Emits processed messages to the Qt application.

- **Command Widgets**
  - Generate user commands.
  - Receive processed responses from the Command Worker.
  - Update the user interface.

---

# Communication Flow

The Command Worker communicates with the Robot Software through the Router Service.

Unlike the Camera Worker and Watchdog Worker, communication is bidirectional.

## Sending Commands

```text
Qt Application
       │
       ▼
Command Worker
       │
       ▼
UI_CMD (DEALER)
       │
       ▼
Router Service
       │
       ▼
ROBOT_CMD (DEALER)
       │
       ▼
Robot Software
```

## Receiving Responses

```text
Robot Software
       │
       ▼
ROBOT_CMD (DEALER)
       │
       ▼
Router Service
       │
       ▼
UI_CMD (DEALER)
       │
       ▼
Command Worker
       │
       ▼
Qt Application
```

The communication sequence is as follows:

1. The Qt application generates a command.
2. The Command Worker serializes and sends the command to the Robot Software.
3. The Robot Software processes the command.
4. The Robot Software sends an acknowledgement or response.
5. The Command Worker receives and parses the response.
6. The processed response is forwarded to the Qt application.

---

# 1. Worker Initialization

During application startup, the **ZmqManager** creates the Command Worker and starts its worker thread.

The initialization sequence consists of:

1. Reading the communication configuration.
2. Creating the ZeroMQ context.
3. Creating the DEALER socket.
4. Connecting to the Router Service.
5. Registering as **UI_CMD**.
6. Waiting for incoming messages.

```text
Create Worker
      │
      ▼
Read Configuration
      │
      ▼
Create ZeroMQ Context
      │
      ▼
Create DEALER Socket
      │
      ▼
Connect to Router
      │
      ▼
Register as UI_CMD
      │
      ▼
Receive REGISTER_ACK
      │
      ▼
Ready for Command Communication
```

---

## 1.1 Creating the Worker

The Command Worker is created by the **ZmqManager** during application startup.

```cpp
CommandWorker::CommandWorker(QObject *parent)
    : ZmqWorkerBase(parent)
{
}
```

Since the worker inherits from **ZmqWorkerBase**, it automatically receives the common worker lifecycle interface shared by all communication workers.

---

## 1.2 Reading the Configuration

The worker reads the Router endpoint from the application configuration.

```cpp
const QString host =
    ConfigReader::instance().value(
        "ZMQ",
        "Host",
        "127.0.0.1");

const QString port =
    ConfigReader::instance().value(
        "ZMQ",
        "CommandPort",
        "5555");
```

Example configuration:

```text
Host : 127.0.0.1
Port : 5555
```

The endpoint becomes:

```text
tcp://127.0.0.1:5555
```

This endpoint is used to establish communication with the Router Service.

---

## 1.3 Creating the DEALER Socket

The Command Worker creates a ZeroMQ context followed by a DEALER socket.

```cpp
m_context = zmq_ctx_new();

m_socket = zmq_socket(
    m_context,
    ZMQ_DEALER);
```

The DEALER socket enables bidirectional communication with the Router Service, allowing the worker to both send commands and receive responses.

---

## 1.4 Connecting to the Router

After creating the DEALER socket, the worker establishes a connection with the Router Service.

```cpp
zmq_connect(
    m_socket,
    endpoint.constData());
```

Communication flow:

```text
Command Worker
        │
        ▼
ZeroMQ DEALER Socket
        │
        ▼
Router Service
```

At this stage, the communication channel is established, but the worker is not yet registered.

---

## 1.5 Registering as UI_CMD

Before exchanging command messages, the worker registers itself with the Router Service using the identity **UI_CMD**.

Example registration message:

```json
{
    "id":"REG_UI_CMD",
    "source":"UI_CMD",
    "target":"ROUTER",
    "type":"REGISTER",
    "priority":1,
    "payload":
    {
        "role":"UI_CMD"
    }
}
```

The Router validates the request and responds with:

```json
{
    "type":"REGISTER_ACK"
}
```

Once the acknowledgement is received, the Command Worker is ready to exchange command messages with the Robot Software.

---

## 1.6 Registration Retry Mechanism

If the Router Service is unavailable during application startup, the registration request may fail.

To handle this situation, the Command Worker automatically retries registration until a **REGISTER_ACK** is received.

```text
Send REGISTER
      │
      ▼
REGISTER_ACK Received?
      │
 ┌────┴────┐
 │         │
Yes        No
 │          │
 ▼          ▼
Continue   Wait 5 Seconds
              │
              ▼
        Retry Registration
```

This retry mechanism allows the Command Worker to recover automatically if the Router Service starts after the Qt application, eliminating the need to restart the application.
